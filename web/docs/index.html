Distributed Video Compression Platform

Отказоустойчивая платформа для запуска ffmpeg на нескольких узлах. Планировщик (Scheduler) принимает задания через HTTP API, воркеры забирают их из очереди PostgreSQL и выполняют транскодирование.

MVP Go 1.22 PostgreSQL

Обзор

Проект реализует минимально жизнеспособную распределённую систему обработки видео. Очередь заданий хранится в PostgreSQL и выдаётся через SELECT … FOR UPDATE SKIP LOCKED — без внешнего брокера сообщений.

Scheduler

HTTP API: создание job, регистрация воркеров, выдача задач, финализация результатов.

Worker

Регистрация, heartbeat, polling заданий, запуск ffmpeg, отправка логов.

PostgreSQL

Единственный источник истины: jobs, workers, job_logs, lease TTL.

Архитектура

┌─────────────┐ HTTP API ┌──────────────────┐ │ Client │ ───────────────► │ Scheduler │ │ (curl/API) │ │ cmd/scheduler │ └─────────────┘ └────────┬─────────┘ │ ▼ ┌──────────────────┐ │ PostgreSQL │ │ jobs / workers │ │ job_logs │ └────────┬─────────┘ │ ┌─────────────────────┼─────────────────────┐ ▼ ▼ ▼ ┌───────────┐ ┌───────────┐ ┌───────────┐ │ Worker 1 │ │ Worker 2 │ │ Worker N │ │ ffmpeg │ │ ffmpeg │ │ ffmpeg │ └───────────┘ └───────────┘ └───────────┘

Ключевые принципы

  • At-least-once — возможны повторные выполнения при сбоях воркера или истечении lease.
  • Lease — при выдаче job воркеру устанавливается lease_expires_at (по умолчанию 30 мин).
  • SKIP LOCKED — несколько воркеров могут безопасно конкурировать за задания без блокировок.

Компоненты

cmd/scheduler

Точка входа HTTP-сервера. Подключается к PostgreSQL через DB_DSN, слушает SCHEDULER_ADDR (по умолчанию :8080).

cmd/worker

Точка входа воркера. Регистрируется в scheduler, шлёт heartbeat, параллельно обрабатывает до max_parallel_jobs заданий.

internal/api

HTTP-хендлеры: создание и получение job, регистрация воркеров, heartbeat, выдача job (AcquireJob), финализация (FinishJob).

internal/store

Слой доступа к PostgreSQL: CRUD для jobs и workers, атомарная выдача заданий, запись логов в транзакции.

internal/worker

Клиент scheduler API и основной цикл воркера: polling, запуск ffmpeg, отправка результата.

internal/ffmpeg

Обёртка над exec ffmpeg: сбор stdout/stderr построчно, проверка входного файла, создание директории для выхода.

Жизненный цикл Job

QUEUED ──► RUNNING ──► COMPLETED ▲ │ │ ▼ RETRY ◄──── FAILED (attempt ≥ max_attempts)
СтатусОписание
QUEUEDЗадание создано и ждёт воркера
RUNNINGНазначено воркеру, установлен lease
COMPLETEDУспешно завершено (финальный статус)
FAILEDИсчерпаны попытки (attempt ≥ max_attempts)
RETRYОшибка выполнения, будет повторно поставлено в очередь

При создании job через API статус сразу QUEUED. При выдаче воркеру attempt увеличивается на 1. Статус NEW зарезервирован в схеме БД, но пока не используется в коде.

HTTP API

Базовый URL: http://<scheduler-host>:8080. Все тела запросов — JSON.

Health

GET /health Проверка доступности scheduler 
{ "status": "ok" }

Jobs

POST /jobs Создать job
GET /jobs/{id} Получить job по ID
GET /jobs/{id}/logs Логи stdout/stderr job (по попыткам, в порядке времени)

POST /jobs — тело запроса:

{
  "input_path": "/data/in/video.mp4",
  "output_path": "/data/out/video.mp4",
  "ffmpeg_args": "-c:v libx264 -crf 23",
  "max_attempts": 3
}

Обязательные поля: input_path, output_path, ffmpeg_args. Ответ: 201 Created с объектом job.

GET /jobs/{id}/logs — ответ 200 OK или 404, если job не найден:

{
  "job_id": "uuid",
  "logs": [
    { "id": 1, "ts": "2026-06-24T12:00:00Z", "stream": "stderr", "line": "..." }
  ]
}

При отсутствии логов возвращается пустой массив logs. Записи упорядочены по ts, затем по id.

Workers

POST /workers/register Регистрация или обновление воркера
POST /workers/heartbeat Heartbeat воркера
POST /workers/request-job Запрос следующего job (SKIP LOCKED + lease)
POST /workers/job-result Финализация job и запись логов

POST /workers/request-job:

{ "worker_id": "uuid" }

// Ответ — job или null, если очередь пуста:
{ "job": { ... } }

POST /workers/job-result:

{
  "worker_id": "uuid",
  "job_id": "uuid",
  "success": true,
  "logs": [
    { "stream": "stdout", "line": "..." },
    { "stream": "stderr", "line": "..." }
  ]
}

Быстрый старт

Рекомендуемый способ — Docker Compose. Ниже — локальный запуск без контейнеров.

  1. Поднять PostgreSQL и применить миграцию:
psql "$DB_DSN" -f db/migrations/001_init.sql
  1. Запустить scheduler:
export DB_DSN="postgres://user:pass@localhost:5432/video?sslmode=disable"
export SCHEDULER_ADDR=":8080"
go run ./cmd/scheduler
  1. Открыть документацию: http://localhost:8080/docs/
  2. Запустить воркер (на машине с ffmpeg и доступом к файлам):
export SCHEDULER_URL="http://localhost:8080"
go run ./cmd/worker
  1. Создать job:
curl -s -X POST http://localhost:8080/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "input_path": "/tmp/in.mp4",
    "output_path": "/tmp/out.mp4",
    "ffmpeg_args": "-c:v libx264 -crf 28"
  }' | jq

Docker

Стек из трёх сервисов: PostgreSQL 16, scheduler (HTTP API) и worker (ffmpeg). Схема БД применяется автоматически при первом старте postgres через /docker-entrypoint-initdb.d/.

Структура

Dockerfile                  # multi-stage: targets scheduler, worker
docker-compose.yml          # оркестрация сервисов
.env.example                # шаблон переменных (скопировать в .env)
docker/postgres/
  postgresql.conf           # настройки PostgreSQL в контейнере
docker/worker/
  entrypoint.sh             # права на том /data для non-root worker

Переменные (.env)

ПеременнаяПо умолчаниюОписание
POSTGRES_USERvideoПользователь БД
POSTGRES_PASSWORDvideoПароль БД (смените в prod)
POSTGRES_DBvideoИмя базы
POSTGRES_PORT5432Порт postgres на хосте
SCHEDULER_PORT8080Порт scheduler на хосте
WORKER_MAX_PARALLEL_JOBS1Параллельные job на воркер
WORKER_GPU_AVAILABLEfalseGPU у воркера

Запуск

cp .env.example .env
docker compose up --build -d
docker compose ps

Документация API: http://localhost:8080/docs/ (порт из SCHEDULER_PORT).

Масштабирование воркеров

docker compose up -d --scale worker=3

Все воркеры делят том jobdata, смонтированный в /data — пути в job указывайте относительно него.

Пример job в Docker

# положить исходник в общий том
docker compose cp ./sample.mp4 worker:/data/in.mp4

curl -s -X POST http://localhost:8080/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "input_path": "/data/in.mp4",
    "output_path": "/data/out.mp4",
    "ffmpeg_args": "-c:v libx264 -crf 28"
  }' | jq

Тесты и CI

Unit-тесты не требуют PostgreSQL: HTTP handlers используют интерфейс api.JobStore с mock-реализацией. Integration-тесты (build tag integration) подключаются только к тестовой БД через TEST_DB_DSN (имя базы должно оканчиваться на _test). Dev-база из docker-compose.yml (порт 5432) не используется.

Локальный запуск

# unit-тесты (быстро, без БД)
make test

# integration: поднимает postgres-test на порту 5433, прогоняет тесты, данные в tmpfs
make test-integration

# остановить тестовый postgres
make test-db-down

# полный набор как в CI
make ci

Тестовый PostgreSQL описан в docker-compose.test.yml: пользователь/БД video_test, порт 5433. DSN по умолчанию: postgres://video_test:video_test@127.0.0.1:5433/video_test?sslmode=disable

Пирамида тестов

СлойПакетЧто проверяет
Unit internal/ffmpeg splitArgs, buildArgs, проверка путей in/out
Unit internal/api Валидация HTTP, коды ответов (mock JobStore)
Unit internal/worker HTTP-клиент scheduler (httptest.Server)
Integration internal/store Очередь, SKIP LOCKED, lease, retry/failed, логи
Integration internal/api Smoke: create → register → request-job → result → get/logs

GitHub Actions

Workflow .github/workflows/ci.yml запускается на pull request и push в main/master:

  1. lint — go vet, go mod verify
  2. unit-tests — go test -short ./...
  3. build — сборка всех пакетов и бинарников scheduler, worker
  4. integration-tests — PostgreSQL 16 (service container), go test -p 1 -tags=integration (один пакет за раз — общая БД)
