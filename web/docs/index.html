Distributed Video Compression Platform

Отказоустойчивая платформа для запуска ffmpeg на нескольких узлах. Планировщик (Scheduler) принимает задания через HTTP API; dispatch идёт через RabbitMQ (transactional outbox), воркеры потребляют сообщения и выполняют транскодирование.

MVP Go 1.22 RabbitMQ

Обзор

Проект реализует минимально жизнеспособную распределённую систему обработки видео. Очередь заданий хранится в PostgreSQL и выдаётся через SELECT … FOR UPDATE SKIP LOCKED — без внешнего брокера сообщений.

Scheduler

HTTP API: создание job, регистрация воркеров, выдача задач, финализация результатов.

Worker

Регистрация, heartbeat, polling заданий, запуск ffmpeg, отправка логов.

PostgreSQL

Единственный источник истины: jobs, workers, job_logs, lease TTL.

Архитектура

┌─────────────┐ HTTP API ┌──────────────────┐ │ Client │ ───────────────► │ Scheduler │ │ (curl/API) │ │ cmd/scheduler │ └─────────────┘ └────────┬─────────┘ │ ▼ ┌──────────────────┐ │ PostgreSQL │ │ jobs / workers │ │ job_logs │ └────────┬─────────┘ │ ┌─────────────────────┼─────────────────────┐ ▼ ▼ ▼ ┌───────────┐ ┌───────────┐ ┌───────────┐ │ Worker 1 │ │ Worker 2 │ │ Worker N │ │ ffmpeg │ │ ffmpeg │ │ ffmpeg │ └───────────┘ └───────────┘ └───────────┘

Ключевые принципы

  • Effectively-once — lease renewal, skip готового output и lease_generation снижают дубли transcode; reaper остаётся safety net.
  • Lease — при выдаче job устанавливается lease_expires_at (по умолчанию 30 мин, JOB_LEASE_DURATION). Воркер продлевает lease через POST /workers/renew-lease во время ffmpeg.
  • Reaper — помечает воркеров без heartbeat как DEAD (timeout по умолчанию 50s) и возвращает RUNNING job с истёкшим lease или на DEAD воркере в outbox retry (DISPATCHED) или FAILED.
  • Idempotency — заголовок Idempotency-Key на POST /jobs; воркер пропускает transcode, если output уже валиден.
  • Transactional outbox — dispatch в RabbitMQ атомарно с записью job; relay публикует из job_outbox.

Компоненты

cmd/scheduler

Точка входа HTTP-сервера. Подключается к PostgreSQL через DB_DSN, слушает SCHEDULER_ADDR (по умолчанию :8080), запускает фоновый reaper.

cmd/worker

Точка входа воркера. Регистрируется в scheduler, шлёт heartbeat, параллельно обрабатывает до max_parallel_jobs заданий.

internal/api

HTTP-хендлеры: создание и получение job, регистрация воркеров, heartbeat, выдача job (AcquireJob), финализация (FinishJob).

internal/store

Слой доступа к PostgreSQL: CRUD для jobs и workers, атомарная выдача заданий, запись логов в транзакции, reaper (MarkStaleWorkersDead, ReapOrphanJobs).

internal/reaper

Фоновый цикл в scheduler: помечает воркеров без heartbeat как DEAD, освобождает job с истёкшим lease или назначенные на DEAD воркеров.

internal/worker

Клиент scheduler API и основной цикл воркера: polling, lease renewal, skip готового output, таймаут ffmpeg, запуск transcode, отправка результата с lease_generation.

internal/presets

Реестр именованных профилей транскодирования: резолв preset → ffmpeg_args, валидация output_ext.

internal/ffmpeg

Обёртка над exec ffmpeg: сбор stdout/stderr, проверка входа/выхода, ffprobe для валидации готового output.

internal/storage

Cloudflare R2 (S3-compatible): presigned upload/download, download/upload объектов для воркера.

Жизненный цикл Job

NEW ──► DISPATCHED ──► RUNNING ──► COMPLETED (R2) ▲ │ ▲ │ │ │ (reaper: lease / DEAD worker → outbox retry) │ ▼ │ outbox retry ◄──── FAILED (attempt ≥ max_attempts)
СтатусОписание
NEWR2 job создан, ожидает загрузки входного файла
DISPATCHEDЗадание принято, ждёт/в пути в RabbitMQ (outbox relay)
RUNNINGНазначено воркеру, установлен lease
COMPLETEDУспешно завершено (финальный статус)
FAILEDИсчерпаны попытки (attempt ≥ max_attempts)

Локальный job через POST /jobs сразу получает статус DISPATCHED (outbox → RabbitMQ). R2 job через POST /jobs/init начинается со статуса NEW и переходит в DISPATCHED после POST /jobs/{id}/submit. При claim воркером attempt увеличивается на 1. Reaper переводит RUNNING в outbox retry (DISPATCHED) или FAILED при истечении lease или падении воркера (без увеличения attempt).

HTTP API

Базовый URL: http://<scheduler-host>:8080. JSON для локальных job; multipart/form-data для upload в R2 через POST /jobs.

Аутентификация

При включённой auth (env SCHEDULER_API_KEY_REQUIRED=true или задан хотя бы один ключ) защищённые endpoint'ы требуют API-ключ в заголовке Authorization: Bearer <key> или X-API-Key.

РольПеременнаяМаршруты
ClientSCHEDULER_CLIENT_API_KEY/jobs*, /presets
WorkerSCHEDULER_WORKER_API_KEY/workers/*
AdminSCHEDULER_ADMIN_API_KEY/admin/api/* и все защищённые маршруты

Без ключа: GET /health, GET /ready, GET /docs/, GET /admin/ (UI; API dashboard требует admin-ключ).

401 — ключ отсутствует или неверный; 403 — ключ верный, но роль не подходит для маршрута.

curl -s -X POST http://localhost:8080/jobs \
  -H "Authorization: Bearer $SCHEDULER_CLIENT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"input_path":"/in.mp4","output_path":"/out.mp4","preset":"h264_crf23"}'

Health

Два probe для orchestrator и load balancer: liveness и readiness.

GET /health Liveness — HTTP-процесс жив (без проверки зависимостей)
GET /ready Readiness — ping PostgreSQL и RabbitMQ; HeadBucket R2 (ошибка R2 → degraded, но 200)

Docker healthcheck scheduler использует /health (liveness). Load balancer readiness → /ready. В Kubernetes: liveness → /health, readiness → /ready.

# GET /health — всегда 200, если процесс отвечает
{ "status": "ok" }

# GET /ready — 200 если postgres и rabbitmq ok; r2 fail даёт degraded, но не 503
{
  "status": "ok",
  "checks": {
    "postgres": { "status": "ok", "latency_ms": 3 },
    "rabbitmq": { "status": "ok", "latency_ms": 5 },
    "r2": { "status": "ok", "latency_ms": 42 }
  }
}

Jobs

POST /jobs/init Создать R2 job и получить presigned URL для upload
POST /jobs/{id}/submit Подтвердить upload и поставить job в очередь
POST /jobs Создать job: JSON (локальные пути) или multipart upload (R2)
GET /jobs/{id} Получить job по ID
GET /jobs/{id}/logs Логи stdout/stderr job (по попыткам, в порядке времени)

POST /jobs — тело запроса:

{
  "input_path": "/data/in/video.mp4",
  "output_path": "/data/out/video.mp4",
  "preset": "h264_crf23",
  "max_attempts": 3
}

JSON: обязательные input_path, output_path и preset или ffmpeg_args (storage: "local"). Multipart (R2): поля file, output_ext и preset или ffmpeg_args — scheduler загружает файл в bucket и сразу ставит job в DISPATCHED.

POST /jobs — multipart upload (R2)

Один запрос «отправил файл и забыл». Требует настроенный R2 на scheduler. Content-Type: multipart/form-data.

ПолеОписание
fileВходной файл (видео/аудио)
presetPreset транскодирования (рекомендуется)
ffmpeg_argsАргументы ffmpeg (deprecated, альтернатива preset)
output_extРасширение результата, напр. mp3
max_attemptsОпционально, по умолчанию 3
 
curl -s -X POST http://localhost:8080/jobs \
  -F "file=@/path/to/video.mp4" \
  -F 'preset=mp3_192k' \
  -F 'output_ext=mp3' | jq

# дождаться COMPLETED, затем download_url:
curl -s http://localhost:8080/jobs/JOB_ID | jq '{status, download_url}'

Лимит размера: env MAX_UPLOAD_BYTES (по умолчанию 512 MiB). Таймаут upload: SCHEDULER_UPLOAD_TIMEOUT (по умолчанию 30m).

Cloudflare R2 (object storage)

Если заданы переменные R2_*, scheduler и worker используют R2. Клиент загружает файл напрямую в bucket по presigned URL; воркер скачивает объект, обрабатывает ffmpeg и загружает результат обратно.

POST /jobs/init — тело запроса:

{
  "preset": "mp3_192k",
  "input_filename": "video.mp4",
  "output_ext": "mp3",
  "max_attempts": 3
}

Ответ 201 Created: объект job (status: "NEW"), upload_url, upload_expires_at, bucket.

# 1. Создать job и получить upload URL
curl -s -X POST http://localhost:8080/jobs/init \
  -H "Content-Type: application/json" \
  -d '{"preset":"mp3_192k","input_filename":"video.mp4","output_ext":"mp3"}' | jq

# 2. Загрузить файл в R2 (URL из upload_url)
curl -X PUT "$UPLOAD_URL" --upload-file ./video.mp4

# 3. Подтвердить upload
curl -s -X POST "http://localhost:8080/jobs/$JOB_ID/submit" | jq

# 4. После COMPLETED — download_url в GET /jobs/{id}
curl -s "http://localhost:8080/jobs/$JOB_ID" | jq '.download_url'

Для job со статусом COMPLETED и storage: "r2" ответ GET /jobs/{id} содержит download_url и download_expires_at (presigned GET).

GET /jobs/{id}/logs — ответ 200 OK или 404, если job не найден:

{
  "job_id": "uuid",
  "logs": [
    { "id": 1, "ts": "2026-06-24T12:00:00Z", "stream": "stderr", "line": "..." }
  ]
}

При отсутствии логов возвращается пустой массив logs. Записи упорядочены по ts, затем по id.

Admin Dashboard

Встроенный web UI по адресу /admin/ для мониторинга jobs и workers в стиле CI pipeline (CircleCI-like). Страница публична; данные загружаются через /admin/api/* с admin API-ключом (хранится в localStorage браузера).

Overview

Сводка: running/queued/completed/failed jobs, active workers. Auto-refresh каждые 5 секунд.

Jobs

Таблица jobs: статус, preset, worker, duration, attempt. Drill-down в job с просмотром stdout/stderr логов.

Workers

Статус воркеров, heartbeat, running jobs. На странице worker — список его jobs и просмотр логов выбранной задачи.

Логи job доступны после финализации попытки (POST /workers/job-result). Для running job UI показывает placeholder до завершения.

Генерация API-ключей

Скрипт scripts/gen-api-keys.sh создаёт три случайных ключа через openssl rand. Вывод идёт в stdout — для prod скопируйте значения в secrets manager, не коммитьте их в репозиторий.

./scripts/gen-api-keys.sh
./scripts/gen-api-keys.sh --format=env
./scripts/gen-api-keys.sh --format=export

После смены ключей перезапустите scheduler и worker. Формат --format=env удобен для локального .env; export — для source .env.

Production secrets

В production не используйте файл .env в образе. Платформа деплоя инжектирует те же переменные окружения из secret store (Kubernetes Secrets, Fly/Railway/ECS env, GitHub Actions secrets и т.п.).

  • DB_DSN — managed PostgreSQL с sslmode=require
  • SCHEDULER_API_KEY_REQUIRED=true + ключи из gen-api-keys.sh
  • R2_* — scoped API token Cloudflare R2
  • После ротации секретов — rolling restart scheduler и worker

Массовая отправка jobs

Скрипт scripts/submit-jobs.sh интерактивно запрашивает client API key и число N, затем отправляет N multipart-запросов POST /jobs с файлом testdata/sample.mp4 (fallback: testdata/fixtures/sample.mp4). Требует настроенный R2 на scheduler.

./scripts/submit-jobs.sh
SCHEDULER_URL=http://localhost:8080 ./scripts/submit-jobs.sh

* Для R2-flow нужны вместе: R2_BUCKET, R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY, R2_ENDPOINT — на scheduler и worker.

Быстрый старт

Рекомендуемый способ — Docker Compose. Ниже — локальный запуск без контейнеров.

  1. Поднять PostgreSQL и применить миграцию:
export DB_DSN="postgres://user:pass@localhost:5432/video?sslmode=disable"
make migrate-up
  1. Запустить scheduler:
export DB_DSN="postgres://user:pass@localhost:5432/video?sslmode=disable"
export SCHEDULER_ADDR=":8080"
go run ./cmd/scheduler
  1. Открыть документацию: http://localhost:8080/docs/ или admin dashboard: http://localhost:8080/admin/
  2. Запустить воркер (на машине с ffmpeg и доступом к файлам):
export SCHEDULER_URL="http://localhost:8080"
export SCHEDULER_WORKER_API_KEY="your-worker-key"
go run ./cmd/worker
  1. Создать job:
curl -s -X POST http://localhost:8080/jobs \
  -H "Authorization: Bearer $SCHEDULER_CLIENT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "input_path": "/tmp/in.mp4",
    "output_path": "/tmp/out.mp4",
    "preset": "h264_crf28"
  }' | jq

docker-compose.yml          # оркестрация сервисов
.env.example                # шаблон переменных (скопировать в .env)
docker/postgres/
  postgresql.conf           # настройки PostgreSQL в контейнере
docker/worker/
  entrypoint.sh             # права на том /data для non-root worker

Запуск

cp .env.example .env
docker compose up --build -d
docker compose ps

Документация API: http://localhost:8080/docs/ (порт из SCHEDULER_PORT).

Масштабирование воркеров

docker compose up -d --scale worker=3

Все воркеры делят том jobdata, смонтированный в /data — пути в job указывайте относительно него.

Пример job в Docker

# положить исходник в общий том
docker compose cp ./sample.mp4 worker:/data/in.mp4

curl -s -X POST http://localhost:8080/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "input_path": "/data/in.mp4",
    "output_path": "/data/out.mp4",
    "preset": "h264_crf28"
  }' | jq

Тесты и CI

Unit-тесты не требуют PostgreSQL: HTTP handlers используют интерфейс api.JobStore с mock-реализацией. Integration-тесты (build tag integration) подключаются только к тестовой БД через TEST_DB_DSN (имя базы должно оканчиваться на _test). Dev-база из docker-compose.yml (порт 5432) не используется.

Локальный запуск

# unit-тесты (быстро, без БД)
make test

# integration: поднимает postgres-test на порту 5433, прогоняет тесты, данные в tmpfs
make test-integration

# остановить тестовый postgres
make test-db-down

# полный набор как в CI
make ci

Тестовый PostgreSQL описан в docker-compose.test.yml: пользователь/БД video_test, порт 5433. DSN по умолчанию: postgres://video_test:video_test@127.0.0.1:5433/video_test?sslmode=disable

GitHub Actions

Workflow .github/workflows/ci.yml запускается на pull request и push в main/master:

  1. lint — go vet, go mod verify
  2. unit-tests — go test -short ./...
  3. build — сборка всех пакетов и бинарников scheduler, worker, migrate
  4. integration-tests — PostgreSQL 16 (service container), migrate up, go test -p 1 -tags=integration (включая internal/migrate)

Ограничения и TODO

  • At-least-once семантика — возможны дублирующиеся запуски ffmpeg
  • Логи хранятся в PostgreSQL — не оптимально для очень больших объёмов
  • Фильтрация job по GPU/CPU пока не используется при выдаче
